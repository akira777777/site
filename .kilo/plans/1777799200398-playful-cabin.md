# Backend Optimization Roadmap: Cyber Slots Casino

## Executive Summary
The current application is a frontend-only React SPA (Vite + TS + TanStack Query + GSAP). No backend, database, or persistent APIs exist (legacy Convex removed). All features (leaderboards, superpowers, reservations, slots) use mocks/hardcoded data. This roadmap introduces a full production backend using the preferred stack (NestJS/Hono + Prisma + PostgreSQL + Redis) to enable real user accounts, secure gameplay, real-time features, and scalability. Focus: scalability (horizontal + caching), performance (low latency queries), security (auth, financial integrity), maintainability (modular TS code, CI/CD).

**Current Gaps Identified**:
- No persistence, auth, or server-side logic → client-side only, vulnerable to tampering (e.g., slot wins).
- Zero concurrency handling or transactions.
- No caching, logging, or monitoring.
- API: Mock only (300ms delay in superpowersApi.ts).
- DevOps: Vercel SPA only; no containers, CI/CD pipelines, or HA setup.

## Recommended Tech Stack
- **Framework**: NestJS (modular, decorators, DI, built-in modules for auth/WS/validation) for complex domain; fallback to Hono for ultra-light API layer if edge-focused.
- **ORM/DB**: Prisma (type-safe queries, migrations, relations) + PostgreSQL (ACID compliance for bets/balances).
- **Cache/Queue**: Redis (sorted sets for leaderboards, pub/sub for realtime, BullMQ for job queues).
- **Auth**: JWT + refresh tokens (httpOnly cookies) + optional OAuth (Google/Apple). Use NestJS Auth or better-auth.
- **API Design**: tRPC (end-to-end TS typesafety with frontend TanStack Query) + REST fallback for public endpoints. Consider GraphQL (@nestjs/graphql) for complex relational queries (user stats + leaderboards).
- **Realtime**: NestJS WebSockets or Socket.IO for live win feeds/leaderboard updates.
- **Validation/Errors**: Zod (reuse frontend schemas) + global exception filters.
- **Logging/Monitoring**: Pino + correlation IDs; Sentry for errors; Prometheus/Grafana for metrics.
- **DevOps**: Docker (multi-stage builds) + docker-compose (dev: app+postgres+redis); GitHub Actions (lint/test/build/deploy); Vercel (frontend) + Railway/Fly.io/AWS (backend) or Kubernetes for HA.
- **Other**: Rate limiting, input sanitization; server-side RNG (crypto module).

## Database Schema Recommendations (Prisma)
Core models for efficiency:
- `User` (id, username, email, passwordHash, balance, createdAt, roles[])
- `Bet` (id, userId, gameType, amount, multiplier, winAmount, seed, createdAt) — indexed on userId + createdAt
- `GameSession` (id, userId, startTime, endTime, totalBets)
- `LeaderboardEntry` (userId, score, rank, updatedAt) — use Redis sorted set primary, Postgres for persistence
- `SuperpowerProgress` (userId, superpowerId, level, points)
- `Reservation` (id, userId, email, bonusType, status, createdAt)
- `Transaction` (id, userId, type, amount, reference, createdAt) for audit

**Optimizations**:
- Indexes: Composite on (userId, gameType, createdAt) for query speed.
- Materialized views or triggers for aggregated leaderboards.
- JSONB for flexible game metadata.
- Partition bets table by month for scale.
- Soft deletes + audit columns.
- Enforce constraints (e.g., balance >= 0 via triggers or app logic).

## API & Concurrency Design
- **Endpoints** (tRPC/REST):
  - Auth: /auth/register, /auth/login, /auth/refresh, /auth/me
  - Games: /games/slots/spin (atomic tx: validate balance, RNG, update), /games/history
  - Leaderboard: /leaderboard/global (cached Redis), /leaderboard/user-rank
  - Users: /users/profile, /users/superpowers (progress updates)
  - Reservations: /reservations (email queue)
- **GraphQL Option**: Single query for "userWithStatsAndLeaderboard" to reduce latency vs multiple REST calls.
- **Concurrency Handling**: Prisma `$transaction` for all financial ops (bet + balance update). Optimistic locking via version field. Queue high-volume spins via BullMQ to prevent DB contention. Use row-level locks for critical sections.

## Caching, Performance & Latency Strategies
- **Redis Layers**:
  - Leaderboards: Sorted sets with 60s TTL + background refresh job.
  - User sessions/balances: 5min cache, invalidate on update.
  - Query results: Cache-aside pattern for /leaderboard, recent wins.
- **Query Optimizations**: Use Prisma `select` + pagination (cursor-based). Avoid N+1 with DataLoader or joins. Batch realtime updates.
- **Frontend Integration**: TanStack Query with staleTime=5m, gcTime=10m (already in useSuperpowers.ts). Add React Query Devtools.
- **Reduce Latency**: WebSockets for push (no polling), edge functions for auth, HTTP/2 + compression, CDN for assets.
- **Benchmarks Target**: API P95 <150ms, cached <20ms; 5k+ concurrent users.

## Error Handling, Logging & Security
- **Error Handling**: Centralized NestJS ExceptionFilter (maps to 4xx/5xx + structured JSON). Retry logic in frontend (already has retry:1).
- **Logging**: Structured logs (level, userId, requestId, latency). ELK or Loki stack.
- **Security**:
  - Auth: Short JWT (15m) + refresh (7d), bcrypt hashes, rate limit (100 req/min/user).
  - Authorization: RBAC (player/admin) + middleware guards.
  - Data: Encrypt sensitive (if real $), validate all inputs, CSRF tokens.
  - RNG: Server-only, auditable seeds, no client trust.
  - OWASP: Helmet, CORS whitelist, SQL injection prevention (Prisma handles).
- **Protocols**: JWT/OAuth2; avoid session cookies for stateless scale.

## Microservices, Consistency & Bottlenecks
- **Architecture**: Start as modular monolith (NestJS modules: AuthModule, GamesModule, LeaderboardModule, NotificationModule). Split to microservices only at 100k+ users (e.g., separate game-engine service).
- **Communication**: Internal: EventEmitter or Redis pub/sub. Cross-service: gRPC or async events (BullMQ/Kafka) with saga for distributed tx (e.g., bet complete → notify leaderboard).
- **Data Consistency**: Strong (ACID tx) for bets/balances. Eventual for leaderboards (background aggregation jobs). Idempotency keys for retries.
- **Potential Bottlenecks & Fixes**:
  - High bet volume: Queue processor (separate worker) + DB connection pool tuning.
  - Leaderboard hot path: Redis primary, Postgres secondary.
  - Realtime sync: WebSocket rooms per game type.
  - Auth hot: Redis token blacklist for revocation.
  - Scaling: Read replicas for Postgres, Redis Cluster, horizontal pod autoscaling.

## DevOps & CI/CD Workflows
- **Containerization**: Dockerfile (node:20-alpine, multi-stage: build → prod). docker-compose.yml (backend, db, redis, frontend proxy).
- **CI/CD (GitHub Actions)**:
  - On PR: lint, typecheck (tsc), unit tests (Jest), e2e (Supertest + testcontainers).
  - On main: build Docker, push to registry, deploy (blue-green to staging/prod).
- **Infra**: Terraform for cloud (RDS Postgres, ElastiCache Redis, ECS/EKS or Fly). Vercel rewrites to /api proxy if co-located.
- **HA/DR**: Multi-AZ, daily backups, health endpoints (/health), auto-scaling, chaos testing.
- **Monitoring**: /metrics endpoint, alerts (error rate >1%, latency >500ms, DB connections >80%).
